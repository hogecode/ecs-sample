## 事前準備

### AWSマネジメントコンソールにログイン
1. ブラウザで [AWS マネジメントコンソール](https://console.aws.amazon.com/) を開く
2. AWSアカウント ID を入力 → 次へ
3. ユーザー名とパスワードを入力 → サインイン

### リージョン確認
1. 画面右上の **リージョン名** をクリック
2. **アジアパシフィック（東京）ap-northeast-1** を選択
3. このリージョンですべての確認を行う

### Terraform出力値を取得
```bash
cd terraform
terraform output
```
重要な値：
- `public_alb_dns_name`: ALBのDNS名（メモしておく）
- `ecs_cluster_name`: ECSクラスタ名
- `rds_instance_endpoint`: RDSエンドポイント

## Terraform出力値の確認

**AWSコンソール → CloudFormation → スタック**

1. AWSコンソール右上の **検索バー** をクリック
2. 「CloudFormation」と入力して選択
3. 左ナビゲーションの **スタック** をクリック
4. 作成したスタック名（例：`ecs-sample-prod`）を探してクリック
5. **出力** タブをクリック
   - `PublicALBDNSName`: ALBのDNS名
   - `ECSClusterName`: ECSクラスタ名
   - `RDSEndpoint`: RDSエンドポイント

## VPC・ネットワークの確認

### VPCの確認

**AWSコンソール → VPC**

1. 左ナビゲーションバーの検索で「VPC」と入力
2. **VPC** をクリック
3. **あなたのVPC** セクションで作成したVPCを確認
   - ✅ CIDR: `10.0.0.0/16`（テーブルに表示）
   - ✅ 状態: `available`

### サブネットの確認

**VPC → サブネット**

1. 左ナビゲーションの **サブネット** をクリック
2. **フィルター** ボタンをクリック → 「VPC ID」でフィルタリング
3. 以下のサブネットが表示される確認：
   - **パブリックサブネット** (2個): `10.0.0.0/24`, `10.0.1.0/24`
   - **プライベートサブネット（アプリ層）** (2個): `10.0.10.0/24`, `10.0.11.0/24`
   - **プライベートサブネット（API層）** (2個): `10.0.20.0/24`, `10.0.21.0/24`
   - **プライベートサブネット（DB層）** (2個): `10.0.30.0/24`, `10.0.31.0/24`

### NAT Gatewayの確認

**VPC → NAT Gateways**

1. 左ナビゲーションの **NAT ゲートウェイ** をクリック
2. 以下の情報を確認：
   - **状態**: `available` ✅
   - **Elastic IP**: パブリックIP が割り当てられている ✅
   - **2個存在**: 各AZに1個ずつ ✅

### ルートテーブルの確認

**VPC → ルートテーブル**

1. 左ナビゲーションの **ルートテーブル** をクリック
2. パブリックサブネット用ルートテーブルをクリック
3. **ルート** タブで以下を確認：
   - `10.0.0.0/16` → ローカル
   - `0.0.0.0/0` → Internet Gateway ✅
4. プライベートサブネット用ルートテーブルをクリック
5. **ルート** タブで以下を確認：
   - `10.0.0.0/16` → ローカル
   - `0.0.0.0/0` → NAT Gateway ✅

## ECSクラスタとタスクの確認

### ECSクラスタの確認

**AWSコンソール → ECS**

1. 左ナビゲーション検索で「ECS」と入力
2. **Elastic Container Service** をクリック
3. 左ナビゲーションの **クラスタ** をクリック
4. 作成したクラスタ（例：`ecs-sample-prod`）をクリック
5. **クラスタの詳細** ページで以下を確認：
   - **ステータス**: `ACTIVE` ✅
   - **実行中のタスク**: 0より大きい数字 ✅
   - **ペンディング中のタスク**: `0` ✅

### ECSサービスの確認

**クラスタ詳細ページ → サービス**

1. 上記のクラスタ詳細ページで **サービス** タブをクリック
2. **Next.jsサービス** を探してクリック
3. サービス詳細ページで確認：
   - **ステータス**: `ACTIVE` ✅
   - **実行中のタスク**: 希望するタスク数と同じ ✅
   - **デプロイメント**: 1つだけ表示、ステータス=`PRIMARY` ✅

4. 同様に **Go Serverサービス** も確認

### タスクの確認

**クラスタ詳細ページ → タスク**

1. クラスタ詳細ページの **タスク** タブをクリック
2. 起動済みタスク一覧が表示される
3. 各タスク行で確認：
   - **ステータス**: `RUNNING` ✅
   - **希望する状態**: `RUNNING` ✅
   - **作成時刻**: 最近 ✅

4. タスク ARN をクリックすると詳細表示：
   - **最後のステータス**: `RUNNING` ✅
   - **希望する状態**: `RUNNING` ✅
   - **コンテナ**: 1個以上表示

### CloudWatch Logsの確認

**AWSコンソール → CloudWatch**

1. 左ナビゲーション検索で「CloudWatch」と入力
2. **CloudWatch** をクリック
3. 左ナビゲーションの **ログ グループ** をクリック
4. 以下のロググループが表示されることを確認：
   - `/ecs/nextjs-prod`
   - `/ecs/go-server-prod`

5. 各ロググループをクリック
6. **ログストリーム** にログが出力されていることを確認
7. ログストリーム をクリック して ログイベント を表示
8. 最新のログメッセージが表示されることを確認 ✅

## ALBとターゲットグループの確認

### ALBの確認

**AWSコンソール → EC2**

1. 左ナビゲーション検索で「EC2」と入力
2. **EC2** をクリック
3. 左ナビゲーションの **ロードバランサー** をクリック
4. **Public ALB** を探してクリック（スキーム=`internet-facing`）
5. ロードバランサー詳細で確認：
   - **ステータス**: `active` ✅
   - **スキーム**: `internet-facing` (公開) ✅
   - **DNS名**: メモしておいたDNS名と一致 ✅
   - **可用性ゾーン**: 2個 ✅

### ターゲットグループの確認

**EC2 → ターゲットグループ**

1. 左ナビゲーションの **ターゲットグループ** をクリック
2. **Next.jsターゲットグループ** を探してクリック
3. ターゲットグループ詳細で確認：
   - **プロトコル**: `HTTP` ✅
   - **ポート**: `3000` ✅

4. **ターゲット** タブをクリック
5. 各ターゲット行で確認：
   - **ステータス**: `healthy` ✅（緑色）
   - **説明**: `Target registration in progress` または成功メッセージ ✅

**⚠️ もしUnhealthyなら:**
1. ステータスの詳細メッセージを読む
2. CloudWatch Logsでエラー確認
3. セキュリティグループ設定を確認（後述）

6. 同様に **Go Serverターゲットグループ** も確認

## API動作テスト

### ブラウザでNext.jsアプリケーションにアクセス

1. 新しいブラウザタブを開く
2. 以下URLを入力:
   ```
   http://<ALB_DNS_NAME>
   ```
   （ALB_DNS_NAMEは、メモしておいたPublic ALB DNSに置き換え）

3. ページ読み込みを待つ（30秒程度）
4. **「Hello World API」ページが表示される** ✅
5. ページ内に **「Hello, World!」というメッセージが表示される** ✅
   - これはNext.js→Go Server→API呼び出しの成功を示す

### DevToolsでAPI呼び出しを確認

1. ブラウザで **F12** を押す (DevTools開く)
2. **コンソール** タブをクリック
3. ログメッセージが出力されていることを確認
4. **ネットワーク** タブをクリック
5. ページをリロード（F5）
6. ネットワークリクエスト一覧で確認：
   - `api/hello` リクエスト → ステータス `200` ✅
   - レスポンス: `{"message":"Hello, World!","status":"success"}` ✅

## RDSの確認

### RDSインスタンス状態確認

**AWSコンソール → RDS**

1. 左ナビゲーション検索で「RDS」と入力
2. **Relational Database Service** をクリック
3. 左ナビゲーションの **データベース** をクリック
4. 作成したDBインスタンス（例：`ecs-sample-prod`）をクリック
5. インスタンス詳細で確認：
   - **DBインスタンスのステータス**: `利用可能` ✅
   - **エンジン**: `mysql` または `postgres` ✅
   - **マルチAZ**: `はい` (本番環境) ✅
   - **ストレージ暗号化**: `はい` ✅

### RDSエンドポイント確認

RDSインスタンス詳細ページで確認：
- **エンドポイント**: `ecs-sample-xxx.rds.amazonaws.com` ✅
- **ポート**: `3306` (MySQL) または `5432` (PostgreSQL) ✅

### RDSセキュリティグループ確認

1. RDSインスタンス詳細ページを下にスクロール
2. **VPCセキュリティグループ** セクションをクリック
3. セキュリティグループ詳細ページが開く
4. **インバウンドルール** タブをクリック
5. 以下のルールを確認：
   - **タイプ**: `MYSQL` (3306) または `PostgreSQL` (5432)
   - **ソース**: Go Serverセキュリティグループ ✅
   - **Bastion** セキュリティグループ ✅

### RDSバックアップ確認

RDSインスタンス詳細ページで確認：
- **バックアップ保持期間**: `7` ✅
- **自動バックアップ**: `はい` ✅

## CloudWatchの確認

### CloudWatch ダッシュボード

**CloudWatch → ダッシュボード**

1. 左ナビゲーションの **ダッシュボード** をクリック
2. 環境用ダッシュボード（例：`ecs-sample-prod`）をクリック
3. ダッシュボードが表示される：
   - **ECS メトリクス**: CPU/メモリ使用率グラフが表示される ✅
   - **ALB メトリクス**: リクエスト数、応答時間が表示される ✅
   - **RDS メトリクス**: CPU使用率、接続数が表示される ✅

### CloudWatch アラーム

**CloudWatch → アラーム**

1. 左ナビゲーションの **アラーム** をクリック
2. **すべてのアラーム** タブをクリック
3. 作成されたアラーム一覧を確認：
   - ECS CPU高い
   - ECS メモリ高い
   - RDS CPU高い
   - など

4. **状態が OK** であることを確認 ✅
   - 🟢 OK = 正常
   - 🔴 アラーム = 問題あり
   - ⚪ データ不足 = 初期状態（しばらく待つ）

### CloudWatch Logs Insights でログ検索

**CloudWatch → ロググループ**

1. `/ecs/go-server-prod` ロググループをクリック
2. **Logs Insights** をクリック
3. クエリ入力エリアに以下を入力：
   ```
   fields @timestamp, @message, @level
   | filter @level like /ERROR/
   | stats count() by @level
   ```
4. **実行** ボタンをクリック
5. **結果** タブでエラーログの統計を確認
6. 0件なら正常 ✅

## 本番環境チェック

### マルチAZ構成確認

**ECS → クラスタ → タスク**

1. ECSクラスタの詳細ページで **タスク** タブをクリック
2. 複数のタスクが表示される確認
3. 各タスク行の **可用性ゾーン** 列を確認
   - `ap-northeast-1a` のタスク 複数個 ✅
   - `ap-northeast-1c` のタスク 複数個 ✅
   - 複数AZに分散している ✅

### セキュリティグループの確認

**EC2 → セキュリティグループ**

1. 左ナビゲーションの **セキュリティグループ** をクリック
2. **ALB セキュリティグループ** をクリック
3. **インバウンドルール** を確認：
   - ポート 80 (HTTP): ソース `0.0.0.0/0` ✅
   - ポート 443 (HTTPS): ソース `0.0.0.0/0` ✅

4. **Next.js セキュリティグループ** をクリック
5. **インバウンドルール** を確認：
   - ポート 3000: ソース `ALB セキュリティグループ` ✅

6. **Go Server セキュリティグループ** をクリック
7. **インバウンドルール** を確認：
   - ポート 8080: ソース `Private ALB セキュリティグループ` ✅
   - ポート 8080: ソース `Next.js セキュリティグループ` ✅

### IAM ロール確認

**IAM → ロール**

1. 左ナビゲーション検索で「IAM」と入力
2. **Identity and Access Management** をクリック
3. 左ナビゲーションの **ロール** をクリック
4. **ecs-sample-task-execution-role** を探してクリック
5. **アタッチされたポリシー** セクションで以下を確認：
   - AmazonEC2ContainerRegistryPowerUser
   - CloudWatchLogsFullAccess
   - など ✅

### 暗号化設定確認

**KMS → カスタマーマネージドキー**

1. 左ナビゲーション検索で「KMS」と入力
2. **Key Management Service** をクリック
3. 左ナビゲーションの **カスタマーマネージドキー** をクリック
4. 作成したKMSキーを探してクリック
5. キー詳細で確認：
   - **キーの状態**: `有効` ✅
   - **説明**: ecs-sample KMS key など ✅

## トラブルシューティング

### ALBターゲットが Unhealthy の場合

**ECS → ターゲットグループ → ターゲット**

1. Unhealthy なターゲットをクリック
2. **説明** 列のメッセージを読む
3. 原因別対応：
   - `Response code mismatch`: CloudWatch Logsでアプリエラーを確認
   - `Target registration in progress`: 30秒待機して再確認
   - `Health checks failed`: セキュリティグループルール確認

**セキュリティグループの確認:**
1. EC2 → セキュリティグループ
2. ターゲットが属するセキュリティグループをクリック
3. **インバウンドルール** でALBからのアクセスが許可されているか確認
4. ルール追加が必要なら **インバウンドルールを編集** をクリック

### ECSタスクが STOPPED の場合

**ECS → クラスタ → タスク**

1. STOPPED タスクをクリック
2. **詳細** セクションで確認：
   - **最後のステータス**: `STOPPED`
   - **理由**: 原因メッセージが表示される

3. 原因別対応：
   - `CannotPullContainerImage`: ECRイメージ確認、IAM権限確認
   - `OutOfMemory`: タスクメモリ割り当て確認
   - エラー含む場合: CloudWatch Logs で詳細確認

### RDS接続エラーの場合

**RDS → データベース → セキュリティグループ確認**

1. RDSインスタンスのセキュリティグループをクリック
2. **インバウンドルール** で以下を確認：
   - Go Server セキュリティグループからのアクセス許可
   - ポート 3306 (MySQL) または 5432 (PostgreSQL)
   - 許可ルールが存在しない場合は **インバウンドルールを編集** で追加

### パフォーマンス遅延の場合

**CloudWatch → ダッシュボード**

1. ダッシュボードで以下を確認：
   - **ECS CPU 使用率**: 70%を超えていないか
   - **ECS メモリ使用率**: 85%を超えていないか
   - **RDS CPU 使用率**: 80%を超えていないか
   - **RDS 接続数**: 上限に近い値でないか
   - **ALB ターゲット応答時間**: 異常に高くないか

2. 問題がある場合：
   - **ECS サービスのスケーリング**:
     1. ECS → クラスタ → サービス をクリック
     2. **更新** ボタンをクリック
     3. **希望する数** を増やす（例: 2 → 4）
     4. **更新** をクリック
   
   - **RDS のスケーリング**:
     1. RDS → データベース → 該当インスタンスをクリック
     2. **変更** ボタンをクリック
     3. **DB インスタンスクラス** を上位に変更
     4. **変更を適用する** をクリック

## チェックリスト
