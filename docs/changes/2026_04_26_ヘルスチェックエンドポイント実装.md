# ヘルスチェックエンドポイント実装

## 概要

ALB（Application Load Balancer）がコンテナのヘルスチェックを実行するために、Next.js と Go Server に health エンドポイントを追加しました。これにより、ECS が不健康なタスクを自動的に置き換えることができます。

## 実装内容

### 1. Go Server Health Endpoint（`server/main.go`）

**エンドポイント:** `GET /health`

```go
// Health Check Endpoint
r.GET("/health", func(c *gin.Context) {
    c.JSON(200, gin.H{
        "status": "healthy",
    })
})
```

**レスポンス:**
```json
{
  "status": "healthy"
}
```

**用途:** Private ALB がこのエンドポイントを定期的にポーリングして、Go Server の稼働状況を確認します。

### 2. Next.js Health Endpoint（`web/app/api/health/route.ts`）

**エンドポイント:** `GET /api/health`

```typescript
export async function GET() {
  return NextResponse.json(
    {
      status: 'healthy',
      timestamp: new Date().toISOString(),
    },
    { status: 200 }
  );
}
```

**レスポンス:**
```json
{
  "status": "healthy",
  "timestamp": "2026-04-26T02:12:20.000Z"
}
```

**用途:** Public ALB がこのエンドポイントを定期的にポーリングして、Next.js の稼働状況を確認します。

## ALB ヘルスチェック設定

### Public ALB（Next.js）

ファイル: `terraform/modules/network/alb/main.tf`

```hcl
health_check = {
  healthy_threshold   = 2          # ヘルシーと判定するまでの連続成功回数
  unhealthy_threshold = 2          # アンヘルシーと判定するまでの連続失敗回数
  timeout             = 5          # ヘルスチェックのタイムアウト（秒）
  interval            = 30         # ヘルスチェック間隔（秒）
  path                = "/api/health"
  matcher             = "200"      # 成功と判定するHTTPステータスコード
}
```

**設定値の説明:**
- `healthy_threshold = 2`: 2回連続で成功したらヘルシー判定
- `unhealthy_threshold = 2`: 2回連続で失敗したらアンヘルシー判定
- `timeout = 5`: 5秒でタイムアウト
- `interval = 30`: 30秒ごとにチェック（合計で最大ヘルシー判定は60秒、アンヘルシー判定は60秒）
- `path = "/api/health"`: Next.js の health エンドポイント
- `matcher = "200"`: HTTP 200 のみを成功と判定

### Private ALB（Go Server）

```hcl
health_check = {
  enabled             = true
  healthy_threshold   = 2
  unhealthy_threshold = 3
  timeout             = 5
  interval            = 30
  path                = "/health"
  matcher             = "200-399"  # 200～399 のステータスコードを成功と判定
  port                = "traffic-port"
}
```

**設定値の説明:**
- `unhealthy_threshold = 3`: 3回連続で失敗したらアンヘルシー判定（より厳密）
- `matcher = "200-399"`: 成功と判定するステータスコード範囲（2xx～3xx）
- `port = "traffic-port"`: コンテナのメイントラフィックポート（8080）を使用

## Blue/Green Deployment との連携

Blue/Green デプロイメント時に、ALB は新しい Green タスクのヘルスチェックを実行します：

1. Green タスク起動
2. ALB が `/health`（Go Server）または `/api/health`（Next.js）をポーリング
3. 2回連続で 200 ステータス → ヘルシー判定
4. トラフィックを Blue から Green に切り替え
5. 古い Blue タスク終了

## ローカルテスト

### Go Server
```bash
curl http://localhost:8080/health
# 出力: {"status":"healthy"}
```

### Next.js
```bash
curl http://localhost:3000/api/health
# 出力: {"status":"healthy","timestamp":"2026-04-26T02:12:20.000Z"}
```

## CloudWatch ログでのモニタリング

ALB のヘルスチェック結果は CloudWatch で確認できます：

```bash
aws logs tail /aws/alb/ --follow
```

アンヘルシーなタスクが検出されると、以下のようなイベントが記録されます：
- `Target failed health checks`
- `Target became unhealthy`
- `Target became healthy`

## トラブルシューティング

### ヘルスチェックが失敗する場合

**1. セキュリティグループの確認**
```bash
# ALB → タスク間のトラフィックが許可されているか確認
aws ec2 describe-security-groups \
  --group-ids sg-xxxxxxxx \
  --query 'SecurityGroups[0].IpPermissions'
```

**2. コンテナログの確認**
```bash
aws logs tail /ecs/ecs-sample-nextjs-dev --follow
aws logs tail /ecs/ecs-sample-go-server-dev --follow
```

**3. ALB ターゲットグループの確認**
```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:...
```

出力例：
```json
{
  "TargetHealthDescriptions": [
    {
      "Target": {
        "Id": "10.x.x.x",
        "Port": 3000
      },
      "HealthCheckPort": "3000",
      "TargetHealth": {
        "State": "healthy",
        "ReasonCode": "N/A"
      }
    }
  ]
}
```
